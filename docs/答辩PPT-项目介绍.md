# 易宿酒店预订平台 · PC 管理端 — 答辩 PPT 项目介绍

> 以下内容可直接用于第五期前端训练营大作业答辩 PPT，按幻灯片分页整理。

---

## 第 1 页：项目名称与定位

**易宿酒店预订平台 — 管理端（PC）**

- **项目性质**：大作业题目中的 **PC 端管理后台** 部分，与移动端用户预订流程配套。
- **目标用户**：**商户**（录入/编辑酒店信息）、**管理员**（审核/发布/下线酒店）。
- **开发周期**：1/29 - 2/26，四周；代码托管于 Git，提交记录体现完整开发过程。

## 第 2 页：技术栈（符合题目要求）

| 层级     | 技术选型           | 说明 |
|----------|--------------------|------|
| **前端** | React + Vite       | 单页应用，组件化开发 |
| **语言** | JavaScript (ESM)   | 模块化与清晰的项目结构 |
| **UI**   | Ant Design 5       | 表单、表格、布局、反馈组件 |
| **路由** | React Router 7     | 登录/注册、商户工作台、管理员审核台 |
| **请求** | Axios              | 统一封装、拦截器、错误与 code 处理 |
| **服务端** | Node.js（独立项目） | 提供 RESTful API，与前端分离部署 |

- 允许使用开源 UI 组件库（Ant Design），提升研发效率与界面一致性。

## 第 3 页：功能实现概览（对应题目要求）

### 管理端必须实现功能 — 均已实现

| 题目要求               | 本项目的实现说明 |
|------------------------|------------------|
| **用户登录/注册**      | 登录页、注册页；区分商户与管理员角色；登录后按角色跳转（商户→工作台，管理员→审核列表）。 |
| **酒店信息录入/编辑/修改** | 商户侧：基础信息、图片上传区、房型与价格、周边与标签、优惠说明；支持新建与编辑已有酒店；保存后通过接口实时更新。 |
| **审核/发布/下线列表** | 管理员侧：酒店审核列表，支持按状态筛选、关键字搜索、分页；操作：审核通过、不通过（必填原因）、下线/恢复上线；下线为状态切换，可恢复。 |

### 与题目要求的对应关系

- **注册界面可选择角色**：注册页提供商户/管理员角色选择。
- **登录无需选角色，自动判断**：登录接口返回 `user.role`，前端据此跳转。
- **审核状态**：通过 / 不通过 / 审核中（待审核）；不通过时展示原因。
- **下线非虚拟删除，可恢复**：通过「下线」「恢复上线」接口切换状态，数据可恢复。

## 第 4 页：项目亮点（对应评分维度）

### 功能完整度（60%）

- 登录/注册、商户酒店录入与编辑、管理员审核/发布/下线全流程打通。
- 商户端：侧边栏布局、我的酒店列表、新建/编辑页、模板引导（首次使用可参考示例酒店）。
- 管理员端：侧边栏布局、审核列表、筛选与分页、通过/不通过/下线与恢复操作。

### 技术复杂度（10%）

- 前后端分离：PC 端统一请求封装，按 `code/data/meta` 处理；与 Node 后端接口文档对齐。
- 长列表与分页：审核列表支持服务端分页与状态筛选。
- 表单与数据流：酒店编辑页多分区表单、房型与价格结构化录入、日期与城市等字段与后端格式一致。

### 用户体验（10%）

- 登录/注册页：全屏背景图、毛玻璃卡片、琥珀黄主色，视觉统一。
- 布局：商户端与管理员端均采用侧边栏 + 内容区，导航清晰。
- 反馈：请求 loading、成功/失败 message、审核不通过必填原因，操作可预期。

### 代码质量（10%）

- 项目结构清晰：`pages`、`layouts`、`services`、`constants`、`router` 分层明确。
- 接口与类型：`auth` / `hotel` 服务与 API 文档一一对应；常量与状态与后端枚举一致。
- 编码规范：ESLint、统一错误处理与请求封装，便于维护与扩展。

### 项目创新性（10%）

- **商户端体验**：模板酒店 +「基于模板新建」，降低首次使用门槛；图片上传区（封面 + 轮播）为后续对接上传接口预留。
- **审核流程**：不通过必填原因、下线可恢复，符合业务说明且便于答辩演示。
- **协作与文档**：双人分工文档（`docs/division.md`）、API 文档（`api_pc.md`）与代码同步，体现“文档即约定”。

## 线上演示地址怎么弄

“线上演示地址” = 把前端（或前后端一起）部署到公网，评委点链接就能打开，不用在自己电脑跑项目。

### 方案一：只部署前端（最简单，适合答辩前时间紧）

1. **把前端代码推到 GitHub**（若尚未推送）。
2. **用 Vercel 部署（免费）**：
   - 打开 [vercel.com](https://vercel.com)，用 GitHub 登录。
   - 点击 “Add New” → “Project”，导入你的 PC 端前端仓库。
   - 构建命令填：`npm run build`；输出目录填：`dist`；根目录保持仓库根目录。
   - 部署完成后会得到一个地址，如：`https://xiecheng-pcend-xxx.vercel.app`，这就是**前端线上演示地址**。
3. **接口仍指向本机时**：线上页面会请求 `http://localhost:3000/api`，评委电脑没跑后端会报错。可以二选一：
   - **答辩时说明**：“线上地址仅作界面展示，完整功能请本地运行前后端”；或
   - 把**后端也部署**到一台公网可访问的服务器（见方案二），然后在前端项目里配置环境变量，让线上前端请求后端线上地址。

### 方案二：前端 + 后端都部署（完整线上演示）

1. **后端**：用 Node 项目自带的启动方式，部署到 **Railway / Render / 腾讯云/阿里云** 等，得到后端 API 根地址，例如：`https://your-api.railway.app/api`。
2. **前端**：在 Vercel 项目里配置**环境变量**：
   - 名称：`VITE_API_BASE_URL`
   - 值：`https://your-api.railway.app/api`（替换成你的后端真实地址）
   - 重新部署后，前端会请求该地址，评委打开前端链接即可完整体验登录、列表、审核等。
3. **线上演示地址**：只填前端的 Vercel 地址即可，例如：`https://xiecheng-pcend.vercel.app`。

### 方案三：没有公网后端时的提交写法

若答辩前只部署了前端、后端未上线，提交时可以这样写：

- **线上演示地址**：`https://xxx.vercel.app`（仅前端，供界面与路由演示）
- **说明**：完整功能（登录、审核、列表数据）需在本地同时启动后端服务后访问上述地址，或使用“本地运行”方式。

### 小结

| 情况           | 线上演示地址填什么        | 评委能否完整体验     |
|----------------|---------------------------|----------------------|
| 只部署了前端   | Vercel 给的链接           | 不能（接口会失败）   |
| 前端 + 后端都部署 | 前端的 Vercel 链接      | 能                   |
| 完全不部署     | 填“暂无，仅提供本地运行” | 需在本地跑才能体验   |

推荐：至少把**前端部署到 Vercel**，得到一条固定链接用于提交和展示；若时间允许，再把后端部署并配好 `VITE_API_BASE_URL`，即可提供完整线上演示。

## 使用说明

- 答辩时可按 **功能 → 技术 → 亮点** 顺序讲，每页控制在 1–2 分钟。
- 演示建议路径：**登录（管理员）→ 审核列表筛选与操作 → 退出 → 登录（商户）→ 我的酒店 → 新建/编辑酒店**，覆盖题目要求的全部管理端功能。
- 若评委问到“与移动端的关系”，可说明：本组负责 PC 管理端，移动端由同题其他小组或协作完成，两端共享同一后端接口与数据。
