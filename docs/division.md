### 一、项目概述

- **项目名称**：易宿酒店预订平台 - 管理酒店信息 PC 系统  
- **项目范围（本组负责部分）**：  
  - 仅负责 **PC 端管理后台**：用户登录/注册、酒店信息录入/编辑/修改、酒店信息审核/发布/下线列表与操作  
  - 数据接口可采用 **Node.js 简易服务或 Mock 接口**，重点体现前端工程化、交互流程与数据联动

---

### 二、整体技术方案

- **前端技术栈**：React + TypeScript + React Router + 状态管理（如 Redux/Zustand/Recoil 其一） + UI 组件库（如 Ant Design）  
- **服务端（简易）**：Node.js（Express/Koa 任一），用于提供酒店数据增删改查和审核状态流转的接口（可视为“假后台”）  
- **项目结构**：  
  - `pages`：登录/注册页、酒店信息编辑页、酒店审核列表页  
  - `components`：表单组件、表格组件、通用布局、搜索筛选组件等  
  - `services`：接口封装（酒店、用户、鉴权）  
  - `store`：用户信息、酒店列表、筛选条件、审核状态等

---

### 三、成员分工总览

- **成员 A：整体功能实现 & 代码质量负责人**  
- **成员 B：整体功能实现 & 工程化负责人**

说明：  
- 实际开发中，**A、B 两人都可以开发所有页面和模块**（登录/注册、酒店信息录入编辑、审核发布列表等），不再做“页面 vs 架构”的强隔离分工；  
- 区别主要体现在“谁主导、谁兜底”：A 更关注交互体验与代码质量，B 更关注工程化与数据流设计；  
- 开发方式为 **单分支 + 错峰开发**：统一提交到 `main` 分支，通过约定开发时间段和频繁拉取代码来避免冲突。

---

### 四、成员 A 具体职责（前端页面与交互）

1. **登录/注册模块（PC 端）**
  - 负责 **登录页 / 注册页** 的整体页面实现与交互：
    - 账号密码输入、校验提示、角色选择（商户/管理员）
    - 登录成功后跳转到对应首页（如酒店列表页或审核列表页）
  - 与成员 B 约定登录接口返回结构、错误码规范，并完成前端错误处理展示。
2. **酒店信息录入/编辑/修改页面（商户视角）**
  - 负责实现酒店信息表单页面，包括：  
    - 必须字段：酒店名（中/英）、地址、星级、房型、价格、开业时间等  
    - 可选字段：周边景点/商圈、交通信息、优惠信息等
  - 负责：
    - 表单布局与交互（分步骤多页表单/分组表单等友好交互设计）  
    - 字段校验规则（必填、格式校验、数值区间、日期逻辑等）  
    - “保存草稿 / 提交审核” 按钮行为和提示反馈
  - 调用成员 B 提供的接口，实现：
    - 新建酒店信息  
    - 编辑已有酒店信息  
    - 修改后实时更新到列表（通过全局状态或重新拉取接口）
3. **基础组件与 UI 细节优化**
  - 负责抽取和维护以下前端组件：
    - 通用表单项组件（如中英名称输入、价格与币种组合控件、日期区间控件等）  
    - 通用按钮、对话框（确认发布/下线）等
  - 负责视觉与交互优化：
    - 页面布局、栅格系统使用、响应式或自适配常见分辨率  
    - 表单操作的 Loading、禁用状态、成功/失败提示等
  - 与成员 B 一起评审组件复用程度，确保代码质量和可维护性。
4. **用户体验与项目展示准备**
  - 主导管理后台 PC 端的 **交互流程梳理与 UX 打磨**，提出改进点（如草稿保存、批量操作、搜索历史等）。  
  - 共同参与 README 编写，重点补充：
    - 界面说明与操作流程截图  
    - 使用说明（如何登录不同角色、常见操作演示）
  - 在结营答辩中，主要负责 **PC 前端页面展示与交互流程讲解**。

### 五、成员 B 具体职责（前端架构与服务）

1. **项目初始化与工程化搭建**
  - 由成员 B 发起，A 参与评审，包含：
    - 构建工具（Vite/CRA）选择与搭建  
    - TypeScript 配置、ESLint/Prettier 规范、目录结构设计  
    - 路由结构设计（登录页 / 注册页 / 酒店编辑页 / 审核列表页等）
  - **Git 使用策略（符合本组实际）**：
    - 统一使用 **单一 `main` 分支** 进行开发，不额外维护 `develop` 和 feature 分支；  
    - 约定“**同一时间段只由一人进行开发和提交**”，另一人主要做代码阅读 / Review / 方案讨论，尽量避免并行修改同一文件；  
    - 每次开始编码前先 `pull` 最新代码，结束后小步多次 `commit` + `push`，保证提交历史能体现完整的项目演进过程。
2. **前端状态管理与数据流设计**
  - 负责选择并搭建状态管理方案（如 Redux/Zustand/Recoil），统一管理：
    - 用户登录态和角色信息  
    - 酒店列表数据、筛选条件、分页信息  
    - 审核状态（通过/不通过/审核中）
  - 负责封装接口调用层（`services`）：
    - 用户登录/注册接口封装  
    - 酒店 CRUD 接口、审核状态流转接口封装
  - 与成员 A 一起定义统一的接口返回格式和错误处理策略（如统一的 `request` 封装）。
3. **酒店信息审核/发布/下线列表页面（管理员视角）**
  - 负责实现管理员端关键页面：
    - 酒店审核列表：支持按状态筛选（通过/不通过/审核中）、关键字搜索  
    - 操作按钮：审核通过/不通过（填写不通过原因）、发布、下线/恢复等
  - 负责审核流程逻辑：
    - 审核状态字段设计（如 `pending/approved/rejected/offline` 等）  
    - 下线非删除、可再恢复的逻辑与接口
  - 完成列表分页/排序等基本体验优化。
4. **Node.js 简易服务与 Mock 数据（可选增强）**
  - 使用 Node.js（Express/Koa 任一）搭建简单后端：
    - 用户注册/登录接口（含角色信息）  
    - 酒店信息增删改查接口  
    - 审核状态更新接口
  - 提供初始 Mock 数据（多家酒店样例），方便前端联调与展示。  
  - 保证项目前端可通过一套命令（如 `npm run dev:server` + `npm run dev:web`）本地完整运行。
5. **性能优化与技术亮点**
  - 在 PC 端列表和表单场景中，主导以下技术优化：
    - 列表渲染优化（分页、懒加载、必要时虚拟列表）  
    - 接口请求节流/防抖（如搜索条件输入时）  
    - 组件拆分、Memo/Callback 等避免不必要重渲染
  - 在结营答辩中，主要负责 **技术选型、架构设计、性能优化点的讲解**。

### 六、协作方式与工作流

- **需求与设计阶段（第 1 周）**
  - 两人共同梳理：
    - PC 端整体功能列表  
    - 角色权限矩阵（商户 vs 管理员能做什么）  
    - 页面原型/简单线框图
  - 输出：
    - 功能列表与页面导航结构  
    - 字段定义文档（酒店数据结构、审核状态定义等）
- **开发阶段（第 2–3 周）**
  - 成员 B 完成项目脚手架、路由、状态管理、接口封装、简易服务；  
  - 成员 A 同步开发 UI 与交互页面，并基于成员 B 的接口进行联调；  
  - 通过 Git 分支协作：  
    - 每个功能新建 feature 分支，完成后合并到 `develop`，保持小步提交。
- **联调与测试阶段（第 3–4 周）**
  - 两人共同完成：
    - 商户角色完整流程验证：登录 → 填写/编辑酒店信息 → 提交审核  
    - 管理员角色完整流程验证：登录 → 审核通过/不通过（含原因）→ 发布/下线/恢复
  - 修复 Bug、优化交互、完善 README 与演示脚本。
- **汇报准备（第 4 周）**
  - 成员 A：准备页面展示流程与用户体验亮点说明。  
  - 成员 B：准备技术架构图、数据流图、性能优化与工程化亮点说明。  
  - 共同录制或排练演示（保证两人都参与讲解）。
